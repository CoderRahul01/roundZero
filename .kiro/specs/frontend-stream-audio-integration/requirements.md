# Requirements Document

## Introduction

This feature implements complete Stream.io Video SDK integration in the React frontend to enable real-time audio communication between users and the AI interviewer. The backend already generates audio using ElevenLabs TTS and publishes it to Stream.io WebRTC calls. The frontend currently has placeholder TODO code and doesn't actually join the Stream.io call, preventing users from hearing the AI interviewer.

The integration will enable bidirectional audio streaming: users will hear the AI interviewer's voice through WebRTC audio tracks, and the AI will receive the user's audio for speech-to-text processing.

## Glossary

- **Stream_SDK**: The @stream-io/video-react-sdk library for WebRTC video/audio calls
- **StreamVideoClient**: The main client class from Stream SDK for managing connections
- **Audio_Track**: A MediaStreamTrack containing audio data from WebRTC
- **Call_ID**: Unique identifier for a Stream.io call in format "session_{uuid}"
- **User_Token**: JWT token generated by backend for Stream.io authentication
- **Agent_Participant**: The AI interviewer participant in the Stream.io call with ID "agent"
- **LiveInterviewScreen**: The React component that hosts the interview interface
- **WebSocket**: Existing WebSocket connection for real-time state updates
- **Microphone_Permission**: Browser permission required to access user's microphone

## Requirements

### Requirement 1: Stream SDK Initialization

**User Story:** As a user, I want the frontend to connect to Stream.io, so that I can participate in audio calls with the AI interviewer.

#### Acceptance Criteria

1. WHEN the LiveInterviewScreen component mounts, THE Frontend SHALL retrieve the Stream API key from the backend response
2. WHEN the Stream API key is available, THE Frontend SHALL initialize a StreamVideoClient instance with the API key and user token
3. THE StreamVideoClient SHALL use the user_id from the authentication context
4. IF the StreamVideoClient initialization fails, THEN THE Frontend SHALL display an error message and log the failure details
5. THE Frontend SHALL store the StreamVideoClient instance in a React ref to prevent re-initialization

### Requirement 2: Call Joining

**User Story:** As a user, I want the frontend to automatically join the Stream.io call, so that I can hear the AI interviewer without manual intervention.

#### Acceptance Criteria

1. WHEN the StreamVideoClient is initialized, THE Frontend SHALL create a call reference using the call_id from the backend
2. THE Frontend SHALL use call type "interview" when creating the call reference
3. WHEN the call reference is created, THE Frontend SHALL invoke the join() method to join the call
4. THE Frontend SHALL enable audio during the join operation
5. IF the call join fails, THEN THE Frontend SHALL retry up to 3 times with exponential backoff (1s, 2s, 4s)
6. IF all retry attempts fail, THEN THE Frontend SHALL display an error message to the user

### Requirement 3: Audio Track Subscription

**User Story:** As a user, I want to automatically receive audio from the AI interviewer, so that I can hear questions and feedback during the interview.

#### Acceptance Criteria

1. WHEN the call is joined successfully, THE Frontend SHALL subscribe to participant events
2. WHEN a participant with ID "agent" joins the call, THE Frontend SHALL retrieve the agent's audio tracks
3. THE Frontend SHALL filter for audio tracks (kind === "audio") from the agent participant
4. WHEN an audio track is received, THE Frontend SHALL create an HTMLAudioElement to play the track
5. THE Frontend SHALL set autoplay to true on the audio element
6. THE Frontend SHALL attach the audio track's MediaStream to the audio element's srcObject
7. FOR ALL audio tracks received, playing the track SHALL produce audible sound through the user's speakers

### Requirement 4: Microphone Access

**User Story:** As a user, I want the frontend to request microphone permission, so that the AI interviewer can hear my responses.

#### Acceptance Criteria

1. WHEN joining the Stream.io call, THE Frontend SHALL request microphone permission from the browser
2. THE Frontend SHALL use the call.microphone.enable() method to enable the microphone
3. IF the user denies microphone permission, THEN THE Frontend SHALL display a warning message explaining that audio input is required
4. IF the user grants microphone permission, THEN THE Frontend SHALL display a success indicator
5. THE Frontend SHALL publish the user's audio track to the Stream.io call
6. THE Agent_Participant SHALL receive the user's audio track for speech-to-text processing

### Requirement 5: Connection Status Display

**User Story:** As a user, I want to see the connection status, so that I know when the audio system is ready.

#### Acceptance Criteria

1. WHILE the StreamVideoClient is initializing, THE Frontend SHALL display "Connecting to audio..." status
2. WHEN the call is joined successfully, THE Frontend SHALL display "Audio connected" status
3. WHEN audio tracks are received from the agent, THE Frontend SHALL display "AI audio active" indicator
4. IF the connection is lost, THEN THE Frontend SHALL display "Connection lost - reconnecting..." status
5. THE Frontend SHALL use color coding: blue for connecting, green for connected, red for errors

### Requirement 6: Audio Level Visualization

**User Story:** As a user, I want to see when the AI is speaking, so that I know the audio system is working correctly.

#### Acceptance Criteria

1. WHEN the agent's audio track is playing, THE Frontend SHALL analyze the audio level using Web Audio API
2. THE Frontend SHALL create an AudioContext and AnalyserNode for the audio track
3. THE Frontend SHALL sample the audio level at 60fps using requestAnimationFrame
4. THE Frontend SHALL pass the audio level (0-100) to the AIPresence component
5. THE AIPresence component SHALL visualize the audio level with animated waveforms
6. WHEN the audio level is above 10, THE Frontend SHALL set ai_state to "speaking"

### Requirement 7: Error Handling

**User Story:** As a developer, I want comprehensive error handling, so that users receive helpful feedback when audio issues occur.

#### Acceptance Criteria

1. IF StreamVideoClient initialization fails, THEN THE Frontend SHALL log the error with context and display "Failed to initialize audio system"
2. IF call joining fails after all retries, THEN THE Frontend SHALL display "Unable to join audio call - please refresh"
3. IF microphone permission is denied, THEN THE Frontend SHALL display "Microphone access required for interview"
4. IF no audio tracks are received within 10 seconds, THEN THE Frontend SHALL display a warning "Waiting for AI audio..."
5. IF the WebRTC connection drops, THEN THE Frontend SHALL attempt automatic reconnection
6. FOR ALL errors, THE Frontend SHALL log detailed error information to the console for debugging

### Requirement 8: Cleanup and Disconnection

**User Story:** As a user, I want the audio connection to be properly closed when I end the interview, so that resources are released correctly.

#### Acceptance Criteria

1. WHEN the user clicks "End Session", THE Frontend SHALL call the leave() method on the Stream call
2. WHEN the component unmounts, THE Frontend SHALL disconnect the StreamVideoClient
3. THE Frontend SHALL stop all audio tracks before disconnecting
4. THE Frontend SHALL close the AudioContext used for audio level analysis
5. THE Frontend SHALL remove all event listeners for participant events
6. IF cleanup fails, THEN THE Frontend SHALL log the error but continue with component unmounting

### Requirement 9: Integration with Existing WebSocket

**User Story:** As a developer, I want Stream.io audio to work alongside the existing WebSocket connection, so that state updates and audio work together seamlessly.

#### Acceptance Criteria

1. THE Frontend SHALL maintain both WebSocket and Stream.io connections simultaneously
2. WHEN WebSocket receives an "ai_audio" message, THE Frontend SHALL ignore it (audio comes from Stream.io)
3. WHEN WebSocket receives a "state_change" message, THE Frontend SHALL update the UI state
4. THE Frontend SHALL use WebSocket for state updates and Stream.io for audio streaming
5. IF either connection fails, THEN THE Frontend SHALL handle each failure independently

### Requirement 10: Audio Playback Reliability

**User Story:** As a user, I want reliable audio playback, so that I don't miss any questions or feedback from the AI interviewer.

#### Acceptance Criteria

1. WHEN an audio track is received, THE Frontend SHALL verify the track is not muted
2. THE Frontend SHALL verify the track's readyState is "live" before playing
3. IF an audio track ends unexpectedly, THEN THE Frontend SHALL log a warning and wait for the next track
4. THE Frontend SHALL handle multiple audio tracks from the agent by playing the most recent track
5. THE Frontend SHALL implement a buffer to prevent audio stuttering on slow connections
6. FOR ALL audio playback, the audio SHALL be synchronized with the AI state updates from WebSocket

## Parser and Serializer Requirements

This feature does not require custom parsers or serializers. It uses:
- Stream.io SDK's built-in WebRTC protocol handling
- Browser's native Web Audio API for audio level analysis
- Standard MediaStream and MediaStreamTrack APIs

## Technical Constraints

- Stream.io SDK version: @stream-io/video-react-sdk ^1.32.4 (already installed)
- Browser compatibility: Chrome 90+, Firefox 88+, Safari 14+ (WebRTC support required)
- Audio format: Opus codec via WebRTC (handled by Stream.io)
- Network requirements: UDP ports for WebRTC, fallback to TURN servers
- React version: 19.2.4 (already installed)

## Success Metrics

- Users can hear AI interviewer audio within 2 seconds of joining the call
- Audio connection success rate > 95%
- Audio latency < 500ms from backend TTS generation to user playback
- Zero audio dropouts during normal network conditions
- Microphone permission grant rate > 80%
